# 配置式表操作路由使用指南

## 📋 概述

通过配置文件定义 URL 与表操作的映射关系，无需编写 Controller 代码，即可实现完整的 CRUD 功能。

### ✨ 核心优势

1. **零代码**：只需配置，无需编写 Controller
2. **被 Spring 完整管理**：拦截器、Filter、日志、事务等全部生效
3. **灵活配置**：支持 YAML 和 Java Config 两种方式
4. **易于维护**：所有路由集中在配置文件中

---

## 🚀 快速开始

### 1. 在 application.yml 中配置

```yaml
icloud:
  table-operation:
    # 基础路径（可选，默认为 /table-op）
    base-path: /table-op
    
    # 路由配置列表
    routes:
      # 用户分页查询
      - path: /user/page
        table-name: sys_user
        action: PAGE_QUERY
        request-dto-class: cn.isqing.icloud.dal.condition.SysUserCondition
        response-vo-class: cn.isqing.icloud.dal.entity.SysUser
        
      # 用户插入
      - path: /user/insert
        table-name: sys_user
        action: INSERT
        request-dto-class: cn.isqing.icloud.dal.entity.SysUser
        
      # 用户更新
      - path: /user/update
        table-name: sys_user
        action: UPDATE
        request-dto-class: cn.isqing.icloud.dal.entity.SysUser
        
      # 用户删除
      - path: /user/delete
        table-name: sys_user
        action: DELETE
        request-dto-class: cn.isqing.icloud.dal.entity.SysUser
        
      # 订单分页查询
      - path: /order/page
        table-name: t_order
        action: PAGE_QUERY
        request-dto-class: cn.isqing.icloud.dal.condition.OrderCondition
        response-vo-class: cn.isqing.icloud.dal.entity.Order
```

### 2. 确保 Entity 和 Condition 类存在

```java
// Entity 类
package cn.isqing.icloud.dal.entity;

@Data
@TableName("sys_user")
public class SysUser {
    private Long id;
    private String username;
    private String email;
    // ... 其他字段
}

// Condition 类
package cn.isqing.icloud.dal.condition;

@Data
public class SysUserCondition extends BaseCondition {
    private String username;
    private String email;
    // ... 查询条件字段
}
```

### 3. 确保 Mapper 存在

```java
package cn.isqing.icloud.dal.mapper;

@Mapper
public interface SysUserMapper extends BaseMapper<SysUser> {
}
```

## 📝 配置说明

### TableOperationRoute 配置项

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| path | String | ✅ | URL路径（相对于base-path） |
| table-name | String | ✅ | 数据库表名 |
| action | ActionType | ✅ | 操作类型：PAGE_QUERY/INSERT/UPDATE/DELETE |
| request-dto-class | Class | ❌ | 请求DTO类全限定名 |
| response-vo-class | Class | ❌ | 响应VO类全限定名 |
| require-auth | boolean | ❌ | 是否需要登录验证（预留） |
| require-permission | boolean | ❌ | 是否需要权限验证（预留） |

### ActionType 枚举值

- `PAGE_QUERY`: 分页查询
- `INSERT`: 插入数据
- `UPDATE`: 更新数据
- `DELETE`: 删除数据

## 🔧 工作原理

### 请求处理流程

```
客户端请求
    ↓
Spring MVC DispatcherServlet
    ↓
已注册的 Filter（日志、认证等）
    ↓
TableOperationRouteInterceptor（匹配配置的URL）
    ↓
如果匹配成功：
    ├─ 读取请求体
    ├─ 构建 TableOperationDto
    ├─ 调用 TableOperationService.execute()
    ├─ 执行对应的表操作
    ├─ 发布事件（如果有订阅者）
    └─ 返回结果
    ↓
如果未匹配：
    └─ 继续正常Controller流程
```

### 为什么能被 Spring 管理？

1. **拦截器注册**：通过 `WebMvcConfigurer.addInterceptors()` 注册
2. **完整生命周期**：经过 Spring MVC 的完整处理链
3. **Filter 生效**：在所有 Filter 之后执行
4. **其他 Interceptor 生效**：可以与其他拦截器协同工作
5. **事务支持**：Service 层的事务注解正常工作

## 🎯 高级用法

### 方式二：Java Config 配置

适合需要动态计算路由的场景：

```java
@Configuration
public class DynamicRouteConfig {
    
    @Bean
    public TableOperationRoutesConfig tableOperationRoutesConfig() {
        TableOperationRoutesConfig config = new TableOperationRoutesConfig();
        config.setBasePath("/api");
        
        // 动态添加路由
        config.getRoutes().add(TableOperationRoute.builder()
            .path("/user/page")
            .tableName("sys_user")
            .action(ActionType.PAGE_QUERY)
            .requestDtoClass(SysUserCondition.class)
            .responseVoClass(SysUser.class)
            .build());
        
        return config;
    }
}
```

### 事件订阅

在增删改操作后，会自动发布事件，可以订阅处理：

```java
@Component
@RouteType(
    r1 = TableOperationEventConstants.R1_TABLE_OPERATION,
    r2 = TableOperationEventConstants.R2_INSERT,
    r3 = "sys_user"
)
public class UserInsertSubscriber implements TableOperationEventSubscriber {
    
    @Override
    public void onEvent(TableOperationEventMsg eventMsg) {
        log.info("新用户创建，ID: {}", eventMsg.getDataId());
        // 发送通知、记录日志等
    }
}
```

## ⚠️ 注意事项

1. **包路径规范**：确保 Entity、Condition、Mapper 遵循规范的包路径
   - Entity: `xxx.entity.Xxx`
   - Condition: `xxx.condition.XxxCondition`
   - Mapper: `xxx.mapper.XxxMapper`

2. **类加载**：配置中的类名必须是完整的全限定类名

3. **拦截器顺序**：TableOperationRouteInterceptor 的 order 设置为 1，会在大多数拦截器之前执行

4. **异常处理**：所有异常都会被捕获并返回统一的错误响应

5. **日志记录**：所有操作都会记录日志，包括请求参数和执行结果

## 📊 对比注解方式

| 特性 | 配置式 | 注解式 |
|------|--------|--------|
| 代码量 | 零代码 | 需要写 Controller |
| 灵活性 | 高（可动态配置） | 低（需修改代码） |
| 可维护性 | 集中管理 | 分散在各 Controller |
| 学习成本 | 低 | 中 |
| 适用场景 | 标准 CRUD | 复杂业务逻辑 |
